# 深圳环评报告书快审工作流

## 流程概述

```
步骤1          步骤2          步骤3          步骤4          步骤5          步骤6          步骤7
内容提取   →   章节分割   →   完整性审查   →   全局检查   →   逐章审查   →   汇总去重   →   生成报告
  (串行)         (串行)         (串行)         (单次LLM)     (并发x3)      (串行)         (串行)
```

## 步骤详解

### 步骤1：内容提取

**输入**：DOCX / DOC / PDF 文件
**输出**：`*_full_text.txt`（完整文本）+ `*_tables.json`（表格数据）

**处理**：
1. 如果是 DOC 文件，使用 macOS `textutil` 转换为 DOCX
2. 使用 `python-docx` 提取 DOCX 文本和表格
3. 如果是 PDF，使用 PyMuPDF 提取文本和表格
4. 提取项目基本信息（项目名称、建设单位、评价等级等）

**性能要求**：10万字报告 ≤ 60秒
**工具**：`python-docx`, `pymupdf`, macOS `textutil`

### 步骤2：章节分割

**输入**：`*_full_text.txt`
**输出**：`chapter_001_概述.txt` … `chapter_013_附件与附表.txt`

**处理**：
1. 按一级标题（Heading 1）拆分章节
2. 章节编号从001开始，最多13个标准章节
3. 处理章节标题中的特殊字符

**标准章节结构（HJ 2.1-2016）**：

| 序号 | 章节名 | 关键词 |
|-----|-------|--------|
| 001 | 概述 | 概述, 项目由来, 建设项目由来 |
| 002 | 第一章 总则 | 第一章, 总则, 评价标准, 评价等级 |
| 003 | 第二章 项目概况 | 第二章, 项目概况, 工程组成 |
| 004 | 第三章 工程分析 | 第三章, 工程分析, 工艺流程, 污染源源强 |
| 005 | 第四章 环境现状调查 | 第四章, 环境现状, 现状调查, 环境监测 |
| 006 | 第五章 环境影响预测 | 第五章, 影响预测, 预测评价, 防护距离 |
| 007 | 第六章 环境保护措施 | 第六章, 环保措施, 污染防治, 处理工艺 |
| 008 | 第七章 环境风险评价 | 第七章, 环境风险, 风险评价, 应急预案 |
| 009 | 第八章 经济损益分析 | 第八章, 经济损益, 环境经济, 效益分析 |
| 010 | 第九章 环境管理监测 | 第九章, 环境管理, 监测计划, 竣工验收 |
| 011 | 第十章 结论与建议 | 第十章, 合理性分析, 选址合理, 结论 |
| 012 | 附件与附表 | 附件, 附表, 附录 |

### 步骤3：章节完整性审查

**输入**：各 `chapter_*.txt`
**依据**：HJ 2.1-2016 + DB4403/T 548-2024

**检查项**：
1. 各标准章节是否存在
2. 各标准章节是否有实质性内容（非仅有标题）
3. 深圳地标要求设置"总量控制"专节是否满足

**输出**：
- `findings/章节完整性/check_result.json`
- `findings/章节完整性/完整性报告.md`

### 步骤4：全局检查（一次性LLM）

**输入**：完整文本 + 项目基本信息
**方式**：单次LLM调用

**检查项**：
- 章节完整性
- 法规标准引用
- 公众参与
- 关键数据汇总确认
- 较大缺陷快速预警

**关注**：较大缺陷快速预警，避免遗漏重大问题

### 步骤5：逐章审查（并发）

**输入**：各章节正文 + 本章相关表格 + 本章必查项
**方式**：3章节并发独立审查
**分块**：内容超限时自动分块（每块 ≤8000字符，OVERLAP 3行）

**每章审查逻辑**：
1. 根据章节关键词匹配适用规则（A类/C类/D类）
2. 检查B类底线规则（适用于所有章节）
3. 调用LLM进行语义审查
4. 输出 `findings/findings_{章节编号}_{章节名}.json`

### 步骤6：汇总去重

**输入**：所有 `findings_*.json`
**处理**：
1. 按规则编号去重
2. 按严重程度分级（重大/较大/一般）
3. 合并同类问题

### 步骤7：生成统一报告

**输入**：所有 findings JSON + 项目信息
**输出**：`审查报告_统一版.md`

**报告结构**：
```markdown
# 技术审查意见

## 第一部分：项目基本情况
### 1.1 项目概况表
### 1.2 关键数据汇总

## 第二部分：审核检查情况
### 2.1 章节完整性审核
### 2.2 各章节技术审核

## 第三部分：问题汇总
### 3.1 重大缺陷（不予批准情形）
### 3.2 较大缺陷（必须修改）
### 3.3 一般缺陷（建议修改）

## 第四部分：审核结论
### 4.1 整体评级
### 4.2 修改建议
```

## 审核规则匹配

### 按章节自动匹配规则

| 章节 | 适用规则类别 |
|-----|-------------|
| 概述 | A类, B类, D类（公众参与） |
| 总则 | A类, B类, C类（评价等级、标准） |
| 项目概况 | A类, B类, C类 |
| 工程分析 | B类, C类（源强、清洁生产） |
| 环境现状调查 | B类, C类（监测数据） |
| 影响预测 | B类, C类（预测方法） |
| 环保措施 | B类, C类, D类 |
| 环境风险 | B类, D类 |
| 经济损益 | D类（技术经济论证） |
| 环境管理 | D类（监测计划） |
| 结论 | B类, C类 |
| 附件与附表 | A类, D类 |

## 配置参数

| 参数 | 默认值 | 说明 |
|-----|-------|------|
| `MAX_CHUNK_SIZE` | 8000 字符 | LLM单次输入上限 |
| `MIN_CHUNK_SIZE` | 1000 字符 | 分块最小字符数 |
| `OVERLAP_LINES` | 3 行 | 分块重叠行数 |
| `CONCURRENT_CHAPTERS` | 3 | 逐章审查并发数 |
| `LLM_TIMEOUT` | 120 秒 | 单次LLM调用超时 |

## 中断恢复

如果在步骤5（逐章审查）过程中中断：

1. 检查 `findings/` 目录下已有的 findings 文件
2. 确定未完成的章节
3. 使用 `run_remaining_chapters.py` 继续审查未完成章节
4. 最后执行步骤6和步骤7生成报告

## 质量控制

1. **规则覆盖检查**：确保55条规则都有对应的检查逻辑
2. **分块质量**：确保分块不会切断完整的句子或表格
3. **去重逻辑**：同一条问题在不同章节重复出现时只保留一条
4. **上下文保持**：分块时保留3行重叠，确保LLM能理解上下文
